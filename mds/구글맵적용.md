# 구글 맵 적용 작업 지시서

## 배경

- 현재 기준으로 Task 07까지 완료되어 음성 목적지 입력, 현재 위치/방향 추적, 안내 문구 계산, DAT Display 전송 흐름은 이미 동작한다.
- Task 08 계획 문서도 존재하며, 여기서 확정해야 할 핵심 방향은 `Google Maps 앱 화면 미러링`이 아니라 `휴대폰에서 지도 이미지 또는 미니맵을 생성한 뒤 DAT Display로 보내는 방식`이다.
- 이번 작업은 기존 화살표, 거리, TTS 안내를 유지한 상태에서 지도 정보를 보조적으로 추가하는 구현을 목표로 한다.

## 작업 목표

- 스마트폰 쪽에서 내비게이션 상태를 기반으로 미니맵 이미지를 생성한다.
- 생성된 미니맵을 기존 `NavigationViewModel`의 Display 전송 흐름에 붙여 스마트글래스에 표시한다.
- 지도 생성 실패, API 제한, 렌더링 지연이 있어도 Task 07의 텍스트/화살표/TTS 안내는 계속 유지되게 만든다.

## 구현 원칙

- `Google Maps 앱 화면 캡처`나 `화면 미러링`은 구현 대상에서 제외한다.
- 지도는 주 안내가 아니라 보조 정보로 취급한다.
- 지도 갱신은 실시간 스트리밍이 아니라 저주기 갱신으로 제한한다.
- 렌더링은 UI 스레드에서 직접 처리하지 말고 비동기 파이프라인으로 분리한다.
- 지도 생성 실패 시 기존 안내 UI만 전송하는 fallback을 항상 유지한다.

## 작업 순서

1. 지도 소스를 먼저 확정한다.
   - 1순위 후보: `Google Maps Static API`
   - 2순위 후보: `Google Maps SDK snapshot`
   - 선택 기준: 구현 난이도, API 키 관리, 비용, 오프라인 대응, 캐시 용이성
2. 렌더 요청과 viewport 모델을 정의한다.
   - 예시: `MapViewportState`, `GlassMapRenderRequest`, `GlassMapRenderResult`
   - 포함 정보: 현재 위치, heading, 목적지 또는 다음 maneuver, zoom, 이미지 크기, 경로 polyline
3. 비동기 렌더링 파이프라인을 추가한다.
   - 지도 생성용 전용 renderer/repository 계층을 분리한다.
   - 메모리 또는 디스크 캐시를 붙인다.
   - 위치 15~25m 이동, heading 20~30도 변화, reroute, 다음 회전 접근 시만 갱신하도록 throttle/debounce를 넣는다.
4. 기존 내비게이션 전송 흐름에 연결한다.
   - `NavigationViewModel`에서 현재 내비게이션 상태를 기반으로 렌더 요청을 만든다.
   - `sendContent` 호출 시 기존 화살표/거리 정보와 함께 미니맵 카드 또는 이미지 블록을 조합한다.
5. fallback과 테스트를 마무리한다.
   - 지도 생성 실패 시 기존 Task 07 화면만 유지한다.
   - 단위 테스트, mock location 테스트, 실제 글래스 또는 MockDeviceKit 검증까지 수행한다.

## 꼭 수정할 가능성이 높은 파일/모듈

- `app/src/main/java/com/meta/wearable/dat/externalsampleapps/displayaccess/navigation/NavigationViewModel.kt`
- `app/src/main/java/com/meta/wearable/dat/externalsampleapps/displayaccess/ui/VoiceNavigationScreen.kt`

- `app/src/test/java/com/meta/wearable/dat/externalsampleapps/displayaccess/display/`
- 필요 시 신규 추가:
  - `navigation/map/` 또는 `display/map/` 패키지
  - 지도 renderer, viewport 모델, 캐시 관리 클래스

## 예상 결과

- 스마트글래스 화면에 기존 방향 화살표와 거리 안내가 유지된다.
- 같은 화면 또는 카드 안에 저주기로 갱신되는 미니맵이 함께 표시된다.
- 위치 변화가 작을 때는 지도 재생성이 발생하지 않아 배터리와 전송 부하가 과하지 않다.
- 지도 렌더링이 실패해도 내비게이션 안내 자체는 중단되지 않는다.

## 테스트 방법

- 단위 테스트
  - viewport 계산
  - 갱신 조건 distance/heading/reroute
  - 캐시 hit/miss
- 통합 테스트
  - mock location으로 직진, 회전, reroute 시나리오 재현
  - 지도 갱신 시 `sendContent`가 과도하게 호출되지 않는지 확인
- 기기 테스트
  - 실제 글래스 또는 MockDeviceKit 환경에서 이미지 표시 여부 확인
  - 회전 직전, 직후, 목적지 근처에서 가독성 확인
- 장애 테스트
  - API 실패, timeout, 빈 경로 데이터 상황에서 Task 07 fallback 유지 확인

## 주의사항

- DAT Display는 전체 화면 미러링 용도가 아니므로 지도 이미지는 작고 단순하게 유지한다.
- 지도 API 선택 전에 비용, 호출 제한, 키 노출 방식을 먼저 정리한다.
- 이미지 갱신 주기가 너무 짧으면 발열, 배터리, 세션 안정성 문제가 생길 수 있다.
- 지도만 보고 방향을 해석하게 만들지 말고, 최종 우선순위는 항상 화살표와 거리 안내에 둔다.
- 이번 작업에서는 다른 문서 정리보다 실제 구현에 필요한 모델, 파이프라인, fallback 기준을 먼저 고정한다.
