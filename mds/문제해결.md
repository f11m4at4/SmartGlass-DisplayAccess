# 문제 해결: `Analysis: error: analysis failure`

## 무엇이 있었나
앱에서 사진 촬영은 정상처럼 보였지만, 분석 결과 대신 아래 메시지가 표시되었습니다.

```text
Analysis: error: analysis failure
```

처음에는 서버가 꺼졌다고 생각하기 쉽지만, 이번 경우에는 **서버는 실행 중**이었습니다.

## 왜 이런 일이 생겼나
핵심 원인은 앱이 분석 서버를 `http://` 주소로 호출하고 있었기 때문입니다.

안드로이드는 보안 때문에 기본적으로 **평문 HTTP(cleartext)** 요청을 막을 수 있습니다.  
즉, 서버가 실제로 켜져 있어도 앱에서 그 서버로 요청을 보내는 단계에서 차단될 수 있습니다.

정리하면:

- 카메라 캡처 자체는 성공할 수 있음
- 하지만 분석 API 호출은 실패할 수 있음
- UI에는 둘을 구분하지 않고 `analysis failure`만 보여서 원인을 바로 알기 어려웠음

## 증상
다음과 같은 상황이 함께 보이면 이 문제를 의심할 수 있습니다.

- 앱은 실행되고 사진도 찍히는 것 같음
- 분석 결과만 실패함
- 서버 프로세스는 이미 실행 중임
- 서버 주소가 `http://`로 되어 있음
- UI에는 자세한 오류 대신 `Analysis: error: analysis failure`만 표시됨

## 어떻게 확인하나

## 1. 엔드포인트 설정 확인
앱이 호출하는 분석 서버 주소를 먼저 봅니다.

확인할 것:

- 주소가 `http://`인지 `https://`인지
- IP와 포트가 맞는지
- 에뮬레이터, 실제 기기, 개발 PC 중 어디를 기준으로 한 주소인지

예:

- `http://192.168.x.x:8000/analyze`
- `https://example.com/analyze`

`http://`라면 안드로이드 네트워크 정책에 막힐 가능성을 먼저 생각해야 합니다.

## 2. 서버 도달 가능 여부 확인
서버가 "켜져 있다"와 "앱이 그 서버에 도달할 수 있다"는 다른 문제입니다.

다음 순서로 확인합니다.

1. 서버가 실제로 실행 중인지 확인
2. 같은 네트워크에 연결되어 있는지 확인
3. 브라우저, `curl`, Postman 등으로 해당 주소가 응답하는지 확인
4. 기기에서 접근 가능한 IP를 사용했는지 확인

중요한 점:

- 서버 콘솔이 떠 있다고 끝이 아님
- 앱이 있는 기기에서 그 주소로 접근 가능해야 함

## 3. logcat 확인
이 문제는 **logcat**에서 단서를 가장 잘 찾을 수 있습니다.

특히 아래와 비슷한 메시지를 찾습니다.

- cleartext traffic 관련 경고 또는 에러
- 네트워크 요청 실패
- connection refused
- timeout

이번 사례에서는 서버 장애보다 **안드로이드가 HTTP 요청을 막았을 가능성**이 더 높습니다.

즉, logcat에서 네트워크 정책 또는 cleartext 관련 메시지가 나오면 거의 원인에 가깝습니다.

## 4. 캡처 성공과 분석 요청 실패를 구분
학생들이 자주 헷갈리는 부분입니다.

앱 흐름은 보통 두 단계입니다.

1. 사진을 캡처함
2. 캡처한 이미지를 분석 서버로 전송함

이번 문제는 1번이 아니라 **2번에서 실패**한 것입니다.

그래서:

- 사진이 찍혔다고 해서 분석 서버 호출까지 성공한 것은 아님
- UI가 단순하면 "전체가 실패한 것처럼" 보일 수 있음
- 실제로는 카메라 기능은 정상이고 네트워크 요청만 실패했을 수 있음

## 왜 UI 메시지가 헷갈렸나
`Analysis: error: analysis failure`는 너무 일반적인 메시지입니다.

이 메시지만 보면 아래를 구분할 수 없습니다.

- 서버가 꺼져 있는지
- 주소가 잘못되었는지
- HTTP가 차단되었는지
- 요청은 갔지만 서버 내부 처리에서 실패했는지

즉, **사용자에게는 같은 실패처럼 보이지만 실제 원인은 여러 가지**일 수 있습니다.

## 어떻게 고치나
해결 방법은 두 가지 방향입니다.

## 방법 1. HTTPS 사용
가장 안전한 방법은 분석 서버를 `https://`로 제공하는 것입니다.

장점:

- 안드로이드 보안 정책에 더 자연스럽게 맞음
- 실제 서비스 환경에서도 권장됨

## 방법 2. 안드로이드 앱의 네트워크 정책 설정
개발 단계에서 꼭 `http://`를 써야 한다면, 앱에서 cleartext 트래픽을 허용하도록 설정해야 합니다.

예를 들어:

- 앱 매니페스트 설정 검토
- network security config 적용
- 특정 도메인 또는 개발용 주소만 허용

주의:

- 무조건 전체 HTTP를 열기보다
- 개발에 필요한 범위만 제한적으로 허용하는 편이 좋음

## 다음에 같은 문제가 나오면 확인할 순서

1. UI 오류 문구를 그대로 믿지 말고, 실패 지점을 나눠서 생각한다.
2. 캡처는 성공했는지, 분석 요청은 실패했는지 분리해서 본다.
3. 앱의 분석 서버 주소가 `http://`인지 확인한다.
4. 서버 IP, 포트, 경로가 맞는지 확인한다.
5. 서버가 기기에서 실제로 reachable한지 확인한다.
6. logcat에서 cleartext, network, connection 관련 메시지를 찾는다.
7. 필요하면 HTTPS로 바꾸거나 Android network policy를 수정한다.

## 예방 체크리스트
배포 전이나 실습 전에 아래를 확인하면 같은 문제를 줄일 수 있습니다.

- 분석 서버 주소가 정확한가
- `http://` 사용 여부를 알고 있는가
- Android cleartext 정책 영향을 확인했는가
- 가능하면 `https://`를 사용하고 있는가
- 실기기에서 서버 접근이 가능한가
- 에러 로그를 logcat에서 바로 볼 수 있게 준비했는가
- UI 에러 메시지가 너무 뭉뚱그려져 있지 않은가
- 캡처 단계와 분석 요청 단계를 따로 로그로 남기고 있는가

## 한 줄 정리
이번 오류는 "서버가 살아 있는데도 앱이 분석 요청을 못 보낸" 문제였습니다.  
가장 가능성이 큰 원인은 `http://` 호출이 안드로이드 cleartext 정책에 막힌 것이고, 해결은 **HTTPS 사용** 또는 **앱 네트워크 정책 설정**입니다.
